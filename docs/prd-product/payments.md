## 1. Feature Overview

- Name: Payment Lifecycle Management
- Summary: Manages recurring/scheduled payments, updates, payoff operations, monthly aggregation, and account statement timelines.
- Target user: Financial users.
- Business value: Tracks execution status of each payable/receivable and keeps invoice balances accurate.

## 2. Problem Statement

- Users need operational control of each payment installment and settlement state.
- Without this, due-date execution, balances, and month-level planning are unreliable.

## 3. Current Behavior

- List/filter payments with invoice and tag context.
- Create standalone installment payments (`save_new`) and invoice-linked payments (via invoice/contract flows).
- Retrieve payment details.
- Update payment fields while open.
- Payoff operation closes payment and updates parent invoice; for fixed invoices, creates next-cycle invoice and payments.
- Scheduled endpoint returns paginated due list.
- Statement endpoint computes opening balance, period transactions, and closing balance from done payments.
- Statement anomalies endpoint returns anomaly list generated by heuristics.

## 4. User Flows

### 4.1 Main Flow

1. User lists upcoming/open payments.
2. User pays off a payment.
3. System marks payment done and adjusts invoice totals and next payment date.

### 4.2 Alternative Flows

1. User edits open payment date/value/type.
2. User requests monthly summarized totals by month.
3. User requests statement for a date range.

### 4.3 Error Scenarios

- Payment not found/inactive.
- Attempt to edit/payoff a payment already done.
- Invalid payload/date parsing failures.

## 5. Functional Requirements

- The system must support filtering and pagination of payments.
- The user can update open payments.
- The system must settle payments and update related invoice values.
- The system must provide month aggregates and statement outputs.

## 6. Business Rules

- Payment status: `open` (0) and `done` (1).
- Editing value updates invoice open value accordingly.
- Payoff of fixed invoice creates future invoice/payment cycle.
- Statement includes only `done` payments for balances.

## 7. Data Model (Business View)

- `Payment`: type (credit/debit), due/payment dates, installment index, fixed flag, value, status, reference.
- `Invoice`: parent container for totals and payment status.

## 8. Interfaces

- APIs:
  - `/financial/payment/`, `/financial/payment/new/`, `/financial/payment/month/`
  - `/financial/payment/<id>/`, `/financial/payment/<id>/save`, `/financial/payment/<id>/payoff`
  - `/financial/payment/scheduled`
  - `/financial/payment/statement/`, `/financial/payment/statement/anomalies`

## 9. Dependencies

- Depends on invoice and contract totals logic.
- Depends on anomaly heuristics from payment AI feature module.

## 10. Limitations / Gaps

- Some error responses use non-standard status codes/messages.
- Part of legacy duplicated payment logic still exists in `financial/views.py`.

## 11. Opportunities

- Add explicit idempotency key for payoff endpoint.
- Add payment change history and reason codes.

## 12. Acceptance Criteria

- Given an open payment, when payoff is called, then payment status becomes done and invoice totals update.
- Given a done payment, when update is called, then update is rejected.
- Given date range, when statement is requested, then opening/closing balances and transactions are returned.

## 13. Assumptions

- Frontend distinguishes standalone payments vs invoice-generated payments.
